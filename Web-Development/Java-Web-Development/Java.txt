			Collections:
				// Lists
					// Methods:
						List<<type>> exampleList;

						exampleList.size();

						exampleList.get(<index>); // Gets the <item> at <index>

						exampleList.add(<index>, <item>); // Adds <item> to specified position in List, if the <index> is within current List size. Shifts all items beyond <index> to the right.
						exampleList.add(<item>); // Adds <item> to end of List.

						<type> item = exampleList.remove(<index>) // Remove <item> at <index>. Shifts all items beyond <index> to the left.
						exampleList.remove(<item>) // Remove first occurence of <item> in List.

					// Implementations:
						// ArrayList 
							// Random Access - O(1) Read from any Index.
							// Add or Remove from end of List - O(1) Write to Index end; O(N) Write for any other Index.
							// Good for Cache-Locality, since contiguous Array of Elements.
							// Note: Primitive Arrays are better; ArrayList cannot store Primitives, they must store Objects, which introduces Object-boxing Memory overhead.
							// Note: Dynamically expands the backing Object[] in fixed multiples, adding to some Memory overhead.

						// LinkedList 
							// Access - O(N) Read from any Index.
							// Add or Remove any Element - O(1).
							// Worse Cache-Locality; requires double jump, first to retrieve the Node and then to retrieve the data in the Node.

				// Maps
					// Methods:
						Map<<key-type>, <value-type>> exampleMap;

						exampleMap.size();

						<value-type> value = exampleMap.get(<key>);
						<value-type> value = exampleMap.getOrDefault(<key>, <default-value>); // Tip: Use computeIfAbsent instead to avoid eager evaluation and memory cost

						exampleMap.put(<key>, <value>); // Add <value> to the Map, referenced by <key>. 
						exampleMap.putIfAbsent(<key>, <value>); // Add <value> to the Map only if the <key> does not exist.

						<value-type> value = exampleMap.computeIfAbsent(<key>, k -> <value>); // Run a Lambda for <key> only if <key> is not present or holds a null.
						exampleMap.computeIfPresent(<key>, (k, v) -> <value>) // Run a Lambda for <key> only if <key> is already set.

						<value-type> value = exampleMap.compute(<key>, (k, v) -> <value>); // Combine computeIfAbsent + computeIfPresent
							// Note: If <value> is null, <key> will be removed from the Map.

						exampleMap.remove(<key>);
						exampleMap.remove(<key>, <value>); // Remove the <key> only if it is currently mapped to the <value> specified.

					// Implementations:
						// HashMap
							// Upon Hash Collision, HashMap dynamically changes a crowded Bucket to a LinkedList, followed by a Red-Black Tree. Worse-Case O(log N).
							// Expands Collection Size upon reaching 75% capacity.

						// LinkedHashMap
							// Same Hash Collision Behaviour as HashMap.
							// Retains Insertion / Least Used Ordering, specified in the Constructor.
							// Has more Memory overhead compared to HashMap, as each Node now has two additional Pointers to the previous / next Node.

						// TreeMap

						// ConcurrentHashMap
							// Individual bins are locking, so that unrelated Keys are not blocking each other.

						// synchronized - declare code where only one Thread may access the region within:
						// Note: This will block other Threads
							class Example {
								// Functions-Level:
									// Per Object-instance

										// ... 
									}

										// ...
									}

								// Block-Level:
									// Declaring 'throws InterruptedException' needed if any Method invoked within throws that Exception
									// Tip: Usually Thread.sleep(), <lock>.wait(), <thread>.join()
									public void fn() throws InterruptedException {
										// synchronized needs a Object to track Threads
										synchronized(this) {
											// ...

											// Re-entrant Property allows the same Thread to activate the same Lock
											// synchronized(this)

											// Every Object has a hidden "Intrinsic Lock" Property that implements some Thread coordination Methods:
												// Make the current Thread release the Lock and make it go into Dormant (i.e. Waiting) State
												// Allows other Threads waiting outside the synchronized Block to enter and acquire the Lock
												// Note: Ensure this.wait() is invoked in a while-loop! Threads can spuriously (i.e. randomly) wake up, even without calling this.notify()!
												while(!<condition>) this.wait();

												// ... update condition ...

												// Wakes up one random Dormant Thread waiting to acquire / re-acquire the Lock in this synchronized Block
												// Tip: For Threads that released the Lock via this.wait(), execution will continue after the this.wait() line, NOT from the beginning of the synchronized Block.
												this.notify();

												// Wakes up all Dormant Threads waiting to acquire / re-acquire the Lock in this synchronized Block
												this.notifyAll();
										}
									}
							}

				// Thread Management:
					Thread.onSpinWait: Instructs the CPU to execute the Thread at a slightly slower rate. Good for saving on CPU Resources while waiting for something, and maintaining the ability to react quickly once the thing materializes. See Optimization Techniques section for other uses.

					Thread.yield: Instructs the Scheduler to give the CPU to another Thread that needs it. Good for optimizing CPU Resources during cooperative multithreading tasks.

					Thread.sleep: Pauses Thread, saving CPU Resources. Good for waiting in the span of milliseconds / seconds.

			Optimization Techniques: Focus on optimizing the Hot Paths.
				Just-in-Time Compilation: Required for a dynamic language such as Java. Compare this with Ahead-of-Time compilation.
					Conceptually, JIT Compilation is NOT a trade of compile-time vs run-time cost; some things in a Program would only be knowable at run-time, such as how frequently a method (i.e. all methods in Java are Virtual) is used.
						If the compiler can prove that only one type of Sub Class goes into a polymorphic reference, it can inline the Sub Class into the compiled code, avoiding the need to do a Virtual Table lookup.

						The Java Microbenchmark Harness (JMH) can be used to measure the JIT Compilation overhead.

						Alternatively, GraalVM can be used for AOT Compilation, replacing the default JVM. AOT Compilation can even be fed with typical runtime data to make it optimise the code correctly.

					Warmup-Routine:
						The JVM interprets first, then compiles Hot Methods when encountered during runtime. Running a warm-up routine can minimize this and related hazards (i.e. deoptimization, runtime Class-loading).

				Garbage Collection:
					Heap Memory Allocation:
						Use Primitives over Objects (i.e. Auto-Boxing); the Agrona Library provides Primitive-keyed Collections.

						Object Pooling to stabilize Memory Allocation.

						Avoid Streams and capturing Lambdas.
							Lambdas that capture require instantiation of a new Lambda instance each time the captured variables change, increasing Memory Consumption.
								If it's not possible to avoid capturing lambdas, ensure that they aren't created unnecessarily. For example, for a Map, use get first to limit calls to a subsequent computeIfAbsent.

							Stream internal library methods are shared by all Streams in the Application; so the call-sites see many different lambdas passing through. This makes the API megamorphic, which the JIT Compiler can't optimize for.

						Avoid the Heap for the true Hot States - the GC only focuses on Heap Management.
							Memory-Mapped Files: Operating System feature that maps the content of the File to the Address Space of the Application's Virtual Memory.

							Chronicle Queue Library provides a Append-only Message Log persisted on Disk.

							Cache beats Pointer-chasing Object graphs.

					Collectors:
						ZGC and Shenandoah are Concurrent Collectors with sub-millisecond pauses, at the cost of some throughput and Memory overhead.

						G1 - Default Collector.

						Epsilon is a No-Op Collector; there's no pauses but the Application will eventually need to be restarted (e.g. during a known period of no activity).

				Lock-Free Execution:
					Single-Writer: If each state only has a single modifier, there will be no need for locks and CAS retry-loops. See: LMAX Disruptor.

				Mechanical Symphony:
					Thread.onSpinWait(): Slows the rate of CPU instruction processing for a Thread (i.e. does NOT reduce the CPU allocated time for a Thread, neither does it put a Thread to sleep).
						Prevents Context Switching for a Thread - blocking and waking a Thread has a cost.

						Prevents unnecessary CPU Prediction flushes - flushes causes latency.
							A CPU Core will aggressively guess what the next CPU instructions would be; if it guesses wrong it will flush the Memory Pipeline. Encountering the onSpinWait will force the CPU to slow down and its guessing.

					Pin a Thread to dedicated CPU core - primarily optimizes for L1 and L2 Caches (i.e. per Core cache). See: OpenHFT Affinity Library.
						Risk of starving GC Collection.

						OS configuration might be needed to ensure CPU core is dedicated for the Application Thread, and that no other tasks outside of the Application are given to the CPU Core.
